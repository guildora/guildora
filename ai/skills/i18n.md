# Skill: Internationalization

## Purpose

Keep language handling, translated UI text, and locale persistence aligned with the repository's actual rules.

## When To Use

- adding UI text
- changing locale behavior
- editing landing-page locale handling
- touching bot or CMS localized content

## Relevant Project Areas

- `apps/web/i18n/locales`
- `apps/hub/i18n/locales`
- `apps/hub/utils/locale-preference.ts`
- `apps/hub/app/middleware/locale.global.ts`
- `apps/cms/src/*`
- `apps/bot/src/i18n/messages.ts`

## Rules And Constraints

- keep `de` and `en` in sync
- persist locale before switching UI locale
- hub and landing manage locale routing separately
- do not use raw backend error text as primary translated UI content

## Step-By-Step Orientation

1. Read `docs/i18n-architecture.md`.
2. Check the existing keys in both locale JSON files.
3. If the change touches CMS or bot behavior, inspect the localized fields or message dictionary too.
4. Update tests if locale-resolution behavior changes.

## Docs References

- `docs/i18n-architecture.md`
- `docs/routing-and-navigation.md`
- `docs/subsystems/cms.md`
- `docs/subsystems/bot.md`

## Common Mistakes To Avoid

- switching the UI locale before `/api/profile/locale` succeeds
- introducing English-only strings in admin or moderator pages
- forgetting that the CMS has its own localization layer
